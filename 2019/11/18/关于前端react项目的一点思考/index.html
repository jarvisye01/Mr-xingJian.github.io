关于前端react项目的一点思考
Posted on 2019-11-18   Updated on 2021-05-24 react javascript 前端 项目结构 思考总结

在接手了两个react项目并且完成交付之后，对于前后端分离之后前端项目的构建，以及react有了一点见解

项目介绍

第一个项目是教师上课用的教辅系统，第二个是一个医疗平台的问答系统，虽然都不是很复杂，但是也都有各自的一些难点所在。因为都是用react构建，我也算是对react有了入门级别的了解认知。所以想对前端项目的构建总结一下自己的经验，特别是用react构建的项目。

我们开发的项目都是单页面web，利用react-router这个路由组件库完成路由的设置，因此路由组件在项目中几乎等于是充当一个“目录”的作用，有一点核心的意味在里面。

项目的数据分发是利用redux结合redux-thunk这个库完成，在部分组件中还存在利用state管理的数据。

UI方面利用antd组件库构建完成。

数据请求是使用的fetch库，一个Promise封装的网络请求库。

首先谈一谈现有项目的缺点和不足

一、命名不合理：

我们几乎从来没有认真的对待我们的项目命名工作，这个在软件工程初期就应该好好解决的问题被我们一直疏忽。当然，在我开始做这些项目的时候我也没有注意到这个问题，否则以上项目的命名也不会是这样不堪。

出现这个问题的重要原因就是我们在项目开始的初期没有进行相关的领域建模，我已导论教师端的项目为例。我们在项目初期思考这个领域的业务的时候就应该做好领域建模，词汇表就应该是一个重要的产出工件，“课前预习”，“课后作业”，“学生信息”，“学生报告”，“任务发布”，“学生反馈”，“学生出题”，“导论题库”等等，我只是罗列我暂时想到的，但是我们将这些词汇与我们代码中的词汇进行对比的时候就不难看出我们的命名有多么糟糕，仅仅是可以辨认而已。另外，如果是不同的同学协作开发，混乱的命名就直接奠定了代码的品味和基调。

解决方案：

  1. 构建领域模型，抽取词汇：在项目初期，必须要构建好相关的领域模型，抽取词汇表并记录下来。之后每一次的功能扩展都必须先建模，确定相关词汇并添加到词汇表中后在进行开发。这里的词汇不仅包括静态名词，也包含相关的操作动词。
  2. 确定统一的命名方式：无论是驼峰还是下划线命名都不存在优劣，关键是项目的整体命名必须要统一形式。禁止使用拼音命名法，推荐由团队英语较好的同学负责，结合谷歌翻译。在英文翻译命名完整之后添加到词汇表，构建完整的中英文对照命名。所有的命名由相关项目负责的同学检查。

二、项目结构不合理

项目结构的优劣直接影响了后面进一步扩展开发的难度，这里我先自我反思，以导论教师端为例剖析其不合理的地方。一个合理的项目结构不仅仅易于扩展，在大家协同开发的时候不会出现很多冲突的问题，一个优雅的项目机构目录奠定了这个项目的水平。

跟我我的一点点经验，在react项目中，我仅仅按照我的思考对项目中的一些构件进行分类：

  • 路由组件：比较核心的组件，这里是大量组件引入的地方，也是整个项目打包后js文件最大的来源，是代码分割的起点
  • redux构件：数据管理与分发，控制整个web用的数据流动。目前是利用redux-thunk框架将多个页面的reducer结合成一个整体的reducer，这些小的reducer分配在不同页面的目录之下（及pages的子目录之下），在目前的开发体验下，这种效果不是很好，数据管理不够集中，但是分解reducer的思想是没问题的。
  • 通用构件：即相关的重用率极高的组件，例如搜索，教师端每个表格头部的标题，都是可复用的组件。但是在目前的项目开发中我们很少去关心组件的复用。
  • 通用工具函数：每一个项目中都会需要一些与相关项目无关的工具函数，例如时间转换函数，手机号码，邮箱的格式验证的正则表达式匹配函数等等，这些公共函数都没有进行相关的抽离，这样不利于这个项目的管理，也不利于后面项目对整个工具函数库的复用。这些函数应该被单独管理，甚至是整个团队都应该构建通用的前端工具库。
  • 配置文件：关于配置文件，一般是包括一些常量的保存，还有生产环境和开发环境的配置。在开发中比较重要的是常量的保存，项目中的常量应该抽离出来，在项目构建之初根据构建的领域模型不仅可以确定项目的组件和函数的命名，也能提炼出这些相关的代码所需的常量，并在开发过程中不断扩充。而部署相关的配置文件主要是关于项目部署到服务器的一些配置文件，比如说前端的nginx代理配置，以及后面涉及到容器部署的docker配置文件，我们希望能将服务器上的工作尽量减少，前端也应该有一个目录管理相关的部署配置文件。
  • 网络请求构件：这是前后端分离之后前端与后端唯一的交互方式，整个项目的网络请求以及相关错误的处理，对于fetch这个Promise的进一步封装，与后端确定详细的错误格式，做到统一处理都是这个版本没有做好的。另外开发与部署环境的请求配置也应该分开处理。
  • css文件：样式文件在react中的管理变得非常棘手，因为jsx语法让我们可以对元素灵活的使用style属性，导致css文件有时候不是必须的。但是在公共样式的处理，以及一些特殊情况下我们一样会用到相关的css文件。所以如何整合css文件，并且如何导入、何时导入、何地导入都会影响到打包后的项目，直接影响项目的打包文件大小和首次加载速度。
  • 普通组件：属于各个页面的组件，一般难以复用。
  • 跨域处理构件：在使用create-react-app创建的项目会使用proxy属性进行跨域配置（配置后要重启），但是如果后端的项目不止一个需要配置多个跨域属性是比较麻烦，最好的还是利用nginx处理跨域问题。

再来回头看导论教师端项目的缺陷：

  • common目录中是footer和header组件：首先就不探讨common这个命名问题，显然这两个组件并不是公共组件，他们只会在App组件中出现一次，放在其中完全是名不副实，而且真正的公共组件也没有抽离出来。
  • 我虽然建了config目录，但是里面并没有放配置文件，显然最初的考虑都败给了执行力，好的想法如果不能强制执行也是垃圾。我的常量配置放到了pages/constants目录中去，里面只有一个index.js文件导出了redux相关的常量，以及项目里面相关业务的常量。显然这二者并不是平行，放在一起是完全没有理由的。
  • util工具库也被放到了pages目录中，这又是一个错误，并不是说这样的安排就不能运行，只是说这样不是一个优秀的逻辑结构。util中的func和style分别放置了通用函数和通用样式文件，但是显然函数库跟样式库在逻辑上并不是同级。
  • fetch网络请求目录知识简单封装了get和post，对于请求时间限制，以及后面的错误处理都没有很好的封装。导致整个项目的错误流处理还不够。
  • css文件组织混乱，样式文件几乎无法避免，在react项目中，样式除了有相关的css文件之外，还有很多jsx内嵌的style属性。然而这其中又会出现大量的重复，css文件不集中以及内嵌样式的冗余会使项目变的糟糕，这样的jsx我认为没人会想看。
  • store文件没有集中管理，redux给我们提供了一个很好的想法，那就是将多个reducer放到不同的文件夹，然后整合成一个大的reducer。这样的分散处理并不利于大型项目的构建，因为有时候一个页面会用到另一个页面的store中的数据，在coding的时候不是很方便。另外对于相关action的管理也不是很好，几乎是每次dispatch都会创建一个action，既没有复用，action的相关格式也没有强制，这些都是既不利于项目的扩展和移交。
  • 路由组件没有使用动态导入，会导致打包后的文件过大。router组件的不同路由会导航至不同的组件，这个组件必然会导入大量的其它组件，而在打包的时候webpack会根据这些导入去寻找相关依赖，打包成bundle，结果就是打包后的js文件过大，首次进入网站的时间很长。

解决方案：

  • 公共组件抽离，放到genericComponent文件夹中。
  • 构建项目的全局配置文件globalConfig，其中存放业务所用的常量，以及一些全局的配置。
  • 构建项目的util工具目录，其中存放相关的工具函数与通用正则表达式等等。最终目标是逐渐积累，构建团队的前端工具函数库。
  • fetch的进一步封装，禁止所以所有请求都用post。对于请求参数的组合要灵活设置，不能只有一个json数据请求的配置，在项目不断扩展的时候也可能会有图片、音频、文件等二进制文件或其他类型文件的传输。可以灵活设计前端参数的格式。
  • 样式文件的抽离，构建globalStyle目录，其中存放css文件，以及通用的style属性结构体。在项目中许多时候都会设置居中，左对齐、边距等等通用的属性，完全可以构建通用的style属性供整个项目使用。将样式文件和style抽离之后，项目的可读性大大增加。
  • store文件的整合，分解reducer的思想不变。但是将各个页面的目录之下的store文件夹移动到全局的store之下，并对reducer采用业务逻辑分解，而不是根据页面分解。并且在全局的store目录之下构建action管理目录以及相关的reducer常量管理目录。
  • 路由文件组件动态导入，使用import()结合高阶组件动态导入组件，这样打包后的js文件会被分割成多个很小的js文件，首页加载速度会提升。
