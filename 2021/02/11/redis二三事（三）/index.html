redis二三事（三）
Posted on 2021-02-11   Updated on 2021-08-21 redis c

字典

background

我们常常将redis成为一种键值对数据库，这也就是体现了redis的本质，你甚至可以将redis看作是一个大型的哈希表，我们对于redis的操作基本都是利用相关的key去操作相关的value。

如果你想学习过任何一门编程语言的容器库，那么你肯定接触过哈希表这个东西，Java中的HashMap就是一个非常好的例子。对于一门编程语言来说，一个哈希表中的value的数据类型一般是固定的，例如HashMap<String, Integer>中的value的类型就是整型，这也就意味着我们只能将整型值放入到哈希表中。

所以我们需要关注的第一个点就是redis是如何屏蔽这些类型差异，让之前介绍过的一些redis定义的数据类型都能进入哈希表。

当然，对于哈希表的研究无外乎就是哈希过程、增删改查、碰撞解决、扩容过程这几个大的方面，我接下来就简单的带大家看一下redis中的哈希。

基本概念

每种哈希表的实现都会有一些比较通用的结构，特别是利用拉链来解决冲突的哈希表设计。有些地方可能会将一个哈希节点叫做Node，同时另外的地方可能叫做Entry，我们应该知道这其实都是一个概念。为了明白redis中是怎么实现的，我们就先介绍一下redis中的一些基本概念。

当然，在接触哈希表之前你还是要了解基本的概念，例如哈希函数将一个key映射成一个值，正好对应哈希表中的一个位置等等，这篇博客并不会介绍哈希的工作原理，我只是介绍redis中的哈希表是怎么设计的。

dict

有了上面的dictht之后我们的哈希表基本就有了雏形，但是为了使用方便，redis继续封装了一层，用一个dict结构来表示字典。

这里的两个表格主要是为了扩容方便，因为当一个table中的节点越来越多，拉链就会越来越长，为了缩短拉链的长度减少搜索时间，就需要扩大table的空间，这个过程会有一部分的节点需要转移到新的位置。redis中的扩容过程就是遍历旧的table，对其中的节点进行相关的哈希计算得到新位置，放到新表对应的位置上，最后就会产生一个新的表格，用新表格取代了旧表格就相当于完成了相关的扩容过程。

rehashidx是表示扩容的过程处理到了哪里，redis为了性能并不会一次性完成扩容，下一次会从上一次的位置继续开始。iterators与dict字典的迭代有关，并不是重点。

dictType

dict中还有一个重要的东西就是dictType这个东西，我们知道如果要将一个键值对放入到字典中要经历如下步骤：

  • 通过hash function为key计算hash value
  • 将key和value复制一份组成dictEntry放到相应的位置

这里我们需要了解为什么需要复制一份，因为在C语言中没有泛型模板这种东西，为了使dictEntry中的key和value可以是任何类型，其中存储的其实是指针，而void*又恰好可以代表任意指针。

如果是指针的话就会有一个问题，外部的修改会影响到容器的内部元素，这是我们不希望看到的，所以在元素进入容器的时候我们都是将元素复制一份放进去的。

如果要从字典中查找一个键值对，主要的步骤如下：

  • 通过hash function为key计算hash value
  • 一次比较该条链表上的key是不是相等(hash value相等不代表key是一样的)

这里的一个比较过程也是我们理解哈希表中的关键，想一想Java中的Object.hashCode()和Object.equals()两个方法。

具体实现

其实当我们将dict的结构有了一个全局的把握之后，其相关实现也就不是那么神秘了，常规的增删查改以及dict的扩容。加上redis又是一个单线程的模型，所以整个代码也并没有出现任何多线程相关的东西，所以还是很好的理解的，来看一下相关的一些API吧。

rehash

rehash是每一个哈希表的实现都要注意的东西，那么在redis中rehash是如何实现的呢？之前我们在dict中看到有2个dictht，其中有一个便是为了辅助rehash的。

我们大概知道rehash是将一个哈希表上的元素迁移到另外一个表上的（具体原理不再详细解释），那么这个过程其实也是一个有很多要注意的地方，我就按照我的理解说两个比较重要的点：

  • rehash是不是一次性完成的：可以肯定的是在redis中rehash并不是一次性完成的，我们知道redis是一个单线程模型，如果一次性完成rehash可能会导致每一次事件处理的时间过长，导致redis的性能下降，所以redis的rehash设计是渐进式的rehash
  • rehash过程的数据存取：既然rehash是渐进式的，那么在rehash的这个时间段内，必然存在两个dictht上同时有数据的情况，那么相应的存取函数的实现也必然会体现出这一点

summary

以上基本就是redis中hash的实现了，在redis中整个全局的key和val的存储都是通过这个结构来实现的。通过拉链来解决hash冲突也是非常经典的方法，有了这些理解就可以对hash有一个全局的认识了。
