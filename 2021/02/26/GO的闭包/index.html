GO的闭包

background

Golang作为一门新生代的编程语言，在近10年得到了巨大的发展，目前除了像字节跳动这样的以Golang为主要编程语言的公司，还有诸如腾讯、阿里内部也有部分项目开始用Golang编写。

Golang是兼具传统语言和新生代语言特点的存在，Golang在保留了C程序员习惯的静态类型和指针操作等特点的同时也带来许多新的东西，例如goroutine(go的协程)、无侵入的接口设计、闭包、通道等等。这些东西结合到一起成就了Golang被一部分人喜爱，也被一部分人吐槽。

本人对Golang的学习时间不到一年，只是在实习的时候用Golang做了一段时间的项目，有些东西理解的不是很到位请原谅。为了进一步深入对Golang的学习，我也是决定了本科的毕业设计用Golang来完成，所以这段时间也是在将之前学习的东西一点一点的捡起来。

今天看到了Golang中的闭包这个特性于是不知怎么的来了兴趣，就想深入的了解一下闭包到底是怎么实现的，于是有了这篇Blog，如果有写的不好的请理解！

tip：有些东西需要你有Golang的汇编前置知识，这些前置知识并不是本篇Blog的重点，如果看的时候有困难，请先看最后我推荐的一些内容！

closure基本概念

闭包就是能够读取其他函数内部变量的函数。这是百度百科中对闭包的解释，非常简洁但是不是很好理解。通俗一点来讲就是声明在一个函数内部的函数可以访问外部函数的相关变量，即使外部函数的寿命已经终结。

Golang中的closure

口说无凭，来看看在Golang中实现一个闭包大概是怎么样的。

这里的OutFunc的返回值是一个函数，并且在内部函数中引用了OutFunc的参数，在之后我们调用内部函数会发现它一直持有这个变量。这与我们的基本常识是违背的，因为我们知道当函数返回之后函数栈上的东西也就销毁了，再使用的话是有危险的，但是这里却可以正常的使用。用更加专业的话来说，这个内部函数捕获了这个变量！

这就是闭包的魔力，不过我们根据观察可以看出一些端倪：

  • 内部函数引用的真正的x肯并不是存在栈上，否则OutFunc返回之后就不能在引用
  • 每一个新返回的内部函数引用的x都是不同的，所以我们猜测每个返回的内部函数都有一个新的x与之对应

Context

函数运行是有上下文的，例如以下全局函数在执行的时候就可以访问相关的全局变量等东西，这些就是函数的上下文。那么闭包函数在运行的时候除了全局上下文肯定还包括它捕获的外部函数变量的上下文，在这里我们可以想像成这个结构：

每次返回一个内部函数的时候OutFunc都会构造这样的一个上下文环境，内部函数在执行的时候需要获取这个与自己对应的闭包上下文环境。这样就大致解释了我们之前的两个问题：（1）这个环境是存储堆上而不是栈上，所以可以一直被内部函数引用（2）每个作为返回值的内部函数都会有一个与自己对应的上下文。

在执行内部函数的时候我们假设它可以用某种方式获取这个上下文，那么在执行的时候引用或者是修改它的捕获变量自然是天经地义的事情。

summary

总结一下，Golang在内部创建了一个结构体用于封装内部函数的指针和捕获的变量：

我们之前的猜测其实是正确了一半，f1 := OutFunc(1)这个函数的返回值是一个函数指针，但是在内部其实返回的是一个ptr = *Context指针，有过C语言基础的同学可能就会很清晰了，*ptr的内容完全可以根据我们的意愿解释为一个函数指针或者是一个Context结构体。

所以这个闭包函数的调用过程就是两步：

  • ptr => DX
  • CALL (*ptr)

内部函数通过DX（ptr）自然尽可以引用属于自己的上下文变量！
