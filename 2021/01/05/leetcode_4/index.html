6.leetcode-324. 摆动排序II（2021.01.07 10:39）

这一题比较难受，提交了好多次，还是只写出了一个O(logN)时间的算法，我用的就是最基本的穿插算法，单独处理了一种特殊情况，来看几个例子：

  • [1,2,3,4]，这种直接穿插即可[1,4,2,3]
  • [1,1,2,3]，这种同样可以直接穿插[1,2,1,3]
  • [1,2,3,3]，还是穿插[1,3,2,3]
  • [1,2,2,3], [1,1,2,2,2,3],[3,4,4,4,4,5,5,5]，这几个都不能直接穿插

上述的数组分别得到[2, 3, 1, 2], [2, 3, 1, 2, 1, 2], [4, 5, 4, 5, 4, 5, 3, 4]，答案是正确的！

10.leetcode-338. 比特位计数（2021.01.08 11:13）

这一题用了一个比较巧妙的算法，比较简单，但是首先你得看出其中的规律，来写几个数吧：

发现了什么规律没有，我来告诉你吧，对于一个数n，如果分为奇数和偶数，有如下区别：

  • 偶数：n = (n/2) << 2，之后补0，所以n和n/2的位为1的个数是相等的
  • 奇数：n = (n/2) << 1，之后补1，所以n的位为1的个数比n/2多一个

因此就有了递推公式：

  • ret[0] = 0
  • ret[i] = ret[i/2] + (i % 2)

通过简单地测试发现结果是正确的，并且是线性时间！

14.leetcode-297. 二叉树的序列化和反序列化（2021.01.10 17:01）

序列化和反序列化都是按照层次遍历来进行的，主要是在序列化的过程中有两个小难点：

  • 如何防止序列化后面的多余的null出现；
  • 如何处理反序列化后面的负数；

我的解决方法是找到层次遍历的最后一个节点，在序列化的过程中注意该节点的父节点，该父节点之后的节点必然都没有孩子节点，并且不需要处理null；反序列化重要注意-这个字符的出现！

序列化和反序列化是很重要的一部分，我们最常见的应该就是json的序列化和反序列化，序列化通常是比较好做的，因为我们能够保证自己序列化数据的正确性，反序列化通常是苦难的，因为我们没有办法保证输入数据的正确性，通常这个正确性比较难验证。在实际的工程中，我们甚至无法保证encoder和decoder是我们一些的一对处理器，甚至无法要求是同一种编程语言编写的，按理来说只要数据格式是固定的，任何语言的处理器都能处理，所以这又需要我们的编写的处理器要兼容一些不会带来错误的格式(例如空格)，是一个极大的挑战。

一定要注意序列化和反序列化中的各种细节，耐心一点就能写对！

使用双指针的前提是我们需要对两个数组进行排序，两个指针i和j不断的移动，遵守如下约束：

  • nums1[i] = nums2[j]: i++, j++;
  • nums1[i] < nums2[j]: i++;
  • nums1[i] > nums2[j]: j++;

前提是这两个指针都不能超过自己所指数组的范围。

27.leetcode-372. 超级次方（2021.01.14 19:19）

对于a = c, b = [x, y, z]我们可以总结出如下规律：

  • c ^ (xyz) = [(c ^ 1) ^ z] * [(c ^ 10) ^ y] * [(c ^ 100) ^ x]

所以我们可以通过上一次的c ^ x计算c ^ x * 10，我把这个称为pBase，因此每次相当于只需要计算10次幂，时间是符合的！

另外为了防止整数的溢出，我们还要对取模进行考虑，我们不可以真的把结果算出来再取模，看看如下的规律：

  • (x * y * z) % mod = (x % mod) * (y % mod) * (z % mod)

当x,y,z相等时就是求幂的状态，所以我这里的pow函数就是利用了这个规律，在相乘的过程中不断地取模，防止结果的溢出！

33.leetcode-354. 俄罗斯套娃信封问题（2021.01.19 12:35）

这一题我只想到了一个跟图有关的解法，但是出现了超时，只能通过80/86个用例。主要的思想就是我们可以先将信封构成一个图，如果信封1的长和宽均小于信封2，那么我们就构件一条有向边。

根据图如何去计算相关的最长路径呢？我们可以发现，如果对于有向无环图中的一个节点node，如果有指向这个node的所有节点的最长路径都计算出来了，那么到node的最长路径永远是上游最大值加上一。

可能有点抽象，分为两种情况讨论：

  • node：入度为0，作为起始节点，最长路径（一条有向路径中的节点数）为1
  • node：入度不为0，上游节点的最长路径为[L1, …, Ln]，到node的最长路径为std::max_element([L1, …, Ln]) + 1

如果一个node的上游节点没有完全计算出来，就不能计算node的最长路径！

35.leetcode-365. 水壶问题（2021.01.21 10:05）

这一题是按照一个小规律去求解的，想了快半个小时才总结出思路，加上coding花了快1小时，确实速度还是个大问题。我就把我思考的过程整理一下：假如这里有水壶分别是x和y，并且满足x < y。

我刚开始想的是两种情况：

  • x, 2x, 3x, 4y, ……
  • y - x, y - 2x, y - 3x, y - 4x, ……

这两种实际就是将x的水不断倒入到y中，以及将y中的水不断倒出到x中。

显然这两种情况不能解决所有问题，因为如果y % x !=0，就会有新的情况，我以我碰到的错误解答(5, 34, 6)为例，y不断向x倒水最终可以得到一个4(y % x)，将4倒入到5中，再将34装满水倒入到5中得到了33，这实际上是一个新的y值。

这一次最后得到的是3，可以得到新的ty值也就是32，这样一直等到ty % x == 0，就是最后一次，不可能得到新的ty了。

所以主要的算法思想如下：

  • 1.ty = y
  • 2.mod = ty % x
  • 3.如果存在(ty - z) % x == 0 || (2 * ty - z) % x == 0，直接返回真
  • 4.ty = y - (x - mod)，得到新的ty
  • 5.如果mod为0则跳出循环，不为0回到第2步

(ty - z) % x == 0 || (2 * ty - z) % x == 0，之所以有两个条件是因为还有可能是两个杯子的水相加，也就是说只要存在一个整数n满足：

  • y - n * x = z
  • y - n * x + x = z
  • y - n * x + y = z

满足任意一个条件就可以，合并就是(ty - z) % x == 0 || (2 * ty - z) % x == 0！

所以我把前n = 10的情况在纸上写了一下：

我们会发现一个奇妙的规律，那就是每个位置的灯泡被触碰过的次数是可以计算的，以6为例子，他会被i = [1, 2, 3, 6]这些轮次调整。

总结一下，位置i上的灯泡怎么才会被调整呢？必然是存在x和y使得i = x * y成立，这里我们要求x <= y，因为x和y的顺序不影响。所以只要是i的因子，就必然会有调整i处灯泡的动作。所以我们只需要知道i有多少个不同的因子，就能知道该处的灯泡动了几次，偶数次必然会灭，奇数次就是亮的。

思考到了这里我就再进了一步，i = x * y成立的情况下，如果有下列条件：

  • x != y，x和y的动作就会抵消
  • x == y，此时只会触发一次动作，显然会让偶数变成奇数

所以如果i是一个完全平方数，那么i处的灯泡就会是亮的，整个算法的时间也会比较好！
