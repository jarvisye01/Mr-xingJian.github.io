浅入浅出Docker与虚拟机
Posted on 2020-07-02   Updated on 2020-07-04 Docker Virtual Machine

Docker与Virtual Machine的简单区别

Docker自从诞生以来就吸引了整个IT界的眼球，Docker让我们可以轻松地从操作系统中隔离出一个可以供我们程序运行的环境，我们可以利用镜像文件轻松地完成这个隔离环境的修改、提交和迁移，而在Docker之前，用来实现这一任务的则是我们大名鼎鼎的前辈–虚拟机，通过虚拟机我们可以在现有操作系统之上构造一个隔离级别相当搞的虚拟造作系统，我们的一台主机可以配置成多台虚拟主机，这些虚拟主机之间是完全相互隔离的

Docker与Virtual Machine就像是前浪与后浪，后者具有强大的功能却稍显笨重，而前者则是飞速成长，有着无限希望

这篇博客全当作为一个概念的理解，没什么非常深入的东西，从一个角度去简单理解二者的区别，如有不对之处请见谅

Images and containers

container：仅仅是一个运行的线程，通过封装一些特性来使其与主机上的其它容器隔离，最突出的就是每个容器有自己的文件系统，由一个Docker image提供

image：包含了需要运行一个程序的所有东西，例如代码、依赖、runtime和filesystem等等

Containers and virtual machine

共同点：这二者的目的都是为了打包一个隔离的计算环境，将各种需要用到的组件从操作系统中隔离开来，二者的主要区别在于可扩展性和可移植性上

container：容器以linux原生线程的形式运行，共享主机的kernel，不需要额外的内存，非常的轻量化，对于container我们常用MB描述其大小

virtual machine：在hypervisor的支持下运行一个成熟的guest os并且通过virtual access去使用主机的资源，通常来说vm会消耗比你的代码逻辑多得多的资源，对于vm machine我们一般用GB来描述其大小

要搞清楚这二者的区别，还要弄清楚一个问题就是hypervisor这个东西：

先给出一段VMware对hypervisor的描述：

A hypervisor, also known as a virtual machine monitor, is a process that creates and runs virtual machines (VMs). A hypervisor allows one host computer to support multiple guest VMs by virtually sharing its resources, like memory and processing.

hypervisor亦成为虚拟机监视器（virtual machine monitor，VMM），是用来创建与运行虚拟机的软件、固件或硬件

这里先区分几个概念：

  • host machine：宿主机，通常是我们的个人pc或者是服务器，宿主机最直接的就是为我们提供相关的硬件资源，当然还包括宿主机上面的操作系统等软件
  • guest machine：虚拟机，在我们的host machine上运行的虚拟的机器，他们使用宿主机的资源来作为自身运行的支撑

这样，很容易就引出了hypervisor，hypervisor是连接这二者的一座桥梁，hypervisor对下将宿主机的硬件资源虚拟化，向上对虚拟机提供隔离运行的平台。联想一下平时我们在谈论CPU的时候最多的是什么？就是那一系列的寄存器，这些寄存器中的值组成CPU运行的现场。而内存呢？现代OS一般会采用虚拟内存这一技术，地址映射到实际物理地址需要经过一步虚拟地址。hypervisor会在软件层面为CPU构建数据结构，这样才能模拟出每个vm都有自己的CPU的假象，寻址过程也需要经过vm的虚拟地址->宿主机虚拟地址->宿主机物理地址。正是因为hypervisor在这二者之间做了大量的工作，这一层的抽象使得宿主机的硬件资源并不能百分之百的流动到虚拟机，虚拟化的过程损失了很多性能，这实质上是一个1 + 1 < 2的过程。当然这个模拟的过程实际上非常复杂，博主这里也没有深入研究，也就点到即止

常见的hypervisor分为两种：

  • Bare metal：直接运行在宿主机的硬件之上
  • Hosted：运行在宿主机的操作系统之上

显然Bare metal的性能会优于Hosted，hypervisor和硬件之间少了一层操作系统，hypervisor直接分配硬件资源，效率自然会更高，与此同时需要做的安全等额外工作也更多，这本是OS的工作。个人PC的话一般都是Hosted类型，毕竟你不太可能只用来玩虚拟机

当然hypervisor只是虚拟机会出现性能瓶颈的一个原因，因为虚拟机是需要完全模拟出一个操作系统，这本身一个非常庞大的工作，虚拟机模拟出的OS需要维持本身OS的许多服务，这本身就会有损失（要知道我们的操作系统在提供我们能够感知到的服务的同时也在背后运行着许多服务），虚拟机的各种调度可能要经历虚拟机kernel->hypervisor->宿主机kernel，这可不是一个小的花费

相比于虚拟机的大而全，Docker就是那么的小而美。前面我们已经说过了，Docker的容器是以linux原生process运行的，这之间少了一层hypervisor的中间商，自然没有差价，速度快的飞起。Docker的容器之间的隔离是通过linux系统本身提供的namespace、cgroup、lxc等技术实现的，这一切都使得Docker容器离我们的OS更近，自然也就离我们的CPU和memory更近

传统IT技术与新生代IT技术的交接

vm machine更多的时候是作为传统IT行业的代表，我们把所有的工作都放到一个目录中，我们需要所有的运行组件和依赖才能够让项目运行起来，这个时候一旦vm machine挂掉也就意味着你的业务会全军覆没

container是新兴IT实践的代名词，与之相关的更多是cloud-native，CI/CD和DevOps这些新生事物，在现代的软件开发过程中我们将软件拆分成一个又一个独立的服务，也就是我们常说的微服务，将这些服务与仅跟它有关的依赖打包到容器，各个服务之间相关隔离

Docker靠什么运行

因为Docker并没有像vm一样构造一个完全隔离的操作系统，那么Docker是依靠什么来保持容器之间的隔离，另外OS中最关键的文件，网络等等又是依托什么完成的呢？

  • namespace
  • cgroup
  • linux container
  • network
  • filesystem

By the way
