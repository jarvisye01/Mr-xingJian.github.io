聊一聊web server
Posted on 2020-07-09   Updated on 2020-07-11 web server php

Web Server在后端中起到的作用

现在因为IDE和编程语言的发展，很多的web server要么被IDE帮我们搞定了（宇宙无敌IDEA），要么就是直接集成到了语言的内部（GO），仿佛给我们一种错觉就是我们编写的后端应用只要那一段代码就可以直接运行。其实这是不对的，我们的语言编写的只是处理请求，但是你考虑过http的请求到达服务器之后是谁来接受的吗？

虽然这几年PHP的热度越来越低，但是用一下PHP会强迫你接触一下apache、nginx这样优秀的web server软件是很有利于我们进一步拓宽自己的视野，深入自己的技术栈的。好好倒腾一下，踩一下坑有益于我们的技术进步。

这是与本文关系不大的一篇文章，只是觉得很好：网络是怎样工作的，https://developer.mozilla.org/zh-CN/docs/learn/How_the_Internet_works

web server在硬件和软件两个领域有着不同的概念

  • 硬件：一台存储webpage，文件资源的计算机，这台计算机被接入互联网，可以接收不同客户端的请求，并将这些资源返回，例如你租一台腾讯云的服务器运行你的网站，我们就称这台计算机为一台服务器。server与client相对，这台计算机更多是提供服务，而我们的个人pc更多是请求服务
  • 软件：以一台linux系统的服务器来讲，它本身并不能承载复杂的请求处理，需要运行相关的软件来完成这些复杂请求的处理，例如apache、nginx和lighttpd等等，这些专门为处理request而设计的软件才是真正请求处理者，OS与硬件只是提供基本能力的支持，这时候的server就更加具体了，这些server软件会去计算机中寻找你所请求的相关文件并返回

动态服务器与静态服务器

  • 静态（static web server）：只提供静态资源的服务，简单点来说你仅仅需要一台linux主机加上相关的web server软件就够了，这样你就能够获取该主机之上的图片，静态网页等文件
  • 动态（dynamic web server）：获取动态文件，例如一个php脚本的执行结果，除了linux主机与web server软件，你可能还需要（以php为例）一个可以调用php解释器执行脚本并将结果返回给web server的软件，或许数据库你也需要，以此类推

**Nginx官网对web server软件的解释：

  1. A web server stores and delivers the content for a website – such as text, images, video, and application data – to clients that request it.(为一个website储存并分发内容)

  2. A web server communicates with a web browser using the Hypertext Transfer Protocol (HTTP). (通过http协议与client通信，这里的client主要是browser)

  3. To deliver dynamic content, most web servers support server‑side scripting languages to encode business logic into the communication.(为了动态的分发内容，许多web server软件支持服务端的脚本语言来支持业务逻辑)

  4. A web server might also cache content to speed delivery of commonly requested content. (可以缓存相关内容来加速分发)

  5. A web server can host a single website or multiple websites using the same software and hardware resources, which is known as virtual hosting.(web server软件可以利用虚拟主机，在相同的软硬件资源中托管多个站点)

这篇文章我们重点分析最后虚拟主机和动态内容的分发，这里的脚本语言我们选择PHP来结合分析。

虚拟主机

1.概念：虚拟主机是指在web服务里的一个独立的网站站点，这个站点对应独立的域名（也可能是IP或者端口），具有独立的程序及资源目录，可以独立地对外提供服务供用户访问。怎么讲说的简单一点，你的一台服务器上可能会部署多个应用，在外部看来我们的每一个应用都是一个独立的服务，好像我们的每个应用都是部署在不同的服务器上。

2.虚拟主机的类型：虽然我们是希望我们的应用是独立的，但是一般我们的服务器的IP地址都是固定的，这该怎么解决呢？

  • 基于端口：首先最容易想到的就是端口了，我们只需要把应用挂到不同的端口之上就可以了，例如常见的8080、8081、8082，这样我们的[IP]:[PORT]形式就能区分不同的请求，发到不同的虚拟主机
  • 基于域名：让用户每次都加上一个端口访问总是不那么优雅的，于是我们还有基于域名的方法，你申请了test.com域名之后完全可以添加子域名a.test.com、b.test.com等等，来自不同域名的请求可以被正确的区分并转发到相应虚拟主机，这样你就可以把所有的虚拟主机都设为80端口，这样你在访问的时候就不用添加端口这个累赘了
  • 基于IP：这个首要的前提是你的服务器有多个IP地址，学习过计网应该知道一个设备是可以有多个IP地址的，这个很好理解，用户访问不同的IP自然可以区分发到相应的虚拟主机，但是这个不太实用，一般我们可以通过网站前端的负载均衡来将不同的IP请求映射到不同的服务器

CGI

CGI是通用网关接口的简称（Common GateWay Interface），这是很早的一种为了动态生成网页的协议，通常一次http请求会携带一定的参数，对应一个CGI脚本，这个脚本在收到参数并进行执行之后会生成一个html，最后返回到客户端。

缺陷：传统的CGI协议每收到一个request就会fork一个进程，处理完之后就exit，如此频繁的进程创建和销毁必然会使利用率大大降低，并发能力实在过薄弱。

FastCGI

fastcgi协议是对cgi协议的一次升级，是为了解决cgi协议每一次请求都要fork一个进程的缺陷，有过OS基础的同学应该这道这个问题的严重性。这也是早期协议很容出现的一个问题，没有考虑到资源的池化和复用。例如http1.1就没有考虑到tcp连接的复用，回http2得到改进，这也是计算机领域不断进步的一个标志

有同学可能会问，为什么不在CGI协议的基础上进行资源的池化和服用呢？

简单点说，CGI在协议制定的初期可能就没考虑到类似于池化和复用的性能指标，那么很多问题协议可能就没考虑到，一个简单的点就是如何区分多个request，这是复用必须要考虑的，在CGI中可能就没有这部分的规定，依次类推我们肯定不能在CGI的基础上实现资源的池化和复用。你的单车道变成十车道总不能还是使用原来的交通规则吧？

先区分几个概念：

CGI协议：规定web server与动态语言脚本之间的通信需要我们规定相关的数据格式和通信方式的协议，就像我们的网络协议一样，这是cgi的rfc文档https://tools.ietf.org/pdf/rfc3875.pdf ，rfc中对CGI的定义就是CGI定义了我们用来描述http请求的抽象参数（meta data），和具体的动态脚本程序接口确定某一个具体平台上的script和Http server之间的接口。

web server与CGI协议是什么关系呢？你的Http request会被web server转为CGI request（协议规定的格式），在动态脚本处理完之后web server再将脚本返回的CGI response转为Http response返回给客户端。

CGI脚本：就是我们常说的动态脚本，例如你的php文件，所有的请求参数和环境变量会被CGI脚本进行处理，CGI脚本利用这些参数和变量动态生成相应的html并返回，这样就是实现了一种动态网页，这就是早期实现动态网页的技术。CGI脚本其实并非必须动态语言来写，所有操作系统上可以运行的语言都可以编写，前提是与web server的交互满足CGI协议规范，如果你使用编译好的静态语言编写，甚至可以不用每次加载解释器，效率更高，但是开发效率就另说了。

CGI Process：CGI脚本的执行需要相关的process来执行，这就是CGI进程

CGI模块：每个到达web server的每一个请求，被转为CGI格式的请求之后需要有一个模块去fork CGI进程来处理，这就是CGI模块，遗憾的是Nginx并不支持这样内置的模块，但是这个并不复杂，在CGI协议中这个模块的作用就是fork一个进程，然后执行相关的CGI脚本，推荐看一下这个https://joyqi.com/develop/nginx-run-cgi.html

历史的进步

这里人们慢慢认识到了CGI协议的不足，就像我们认识到http协议的不足。我们希望不同的http request与不同的CGI脚本之间进行交互式可以复用通道，亦如http2可以复用tcp连接。并且，我们不希望每次调用CGI脚本都需要fork一个新的进程，用完就exit，我们要池化这些资源，维护多个进程，有需要时直接取出用完再放回去，避免进程创建和销毁的损耗。这个时候我们肯定会考虑两个事情：

  1. 现有的CGI协议并没有考虑到通道的复用和另外的性能部分，需要升级
  2. CGI Process没有池化的功能，我们需要管理多个CGI Process的工具

下面就看一看Fastcgi是如何解决这两个问题的。

FastCGI：这是FastCGI的文档http://www.mit.edu/~yandros/doc/specs/fcgi-spec.html ，Fastcgi是CGI的一种改进，Fastcgi协议要求通信连接可以复用，并引入了流等等概念。这里博主一开始没搞清楚的就是要支持复用和流这个特性为什么需要协议的支持，因为我们所有的数据都是以协议规范的格式进行交换的，你如果没有改进，多个请求的数据涌入到一个tcp连接中该怎么区分呢？例如Http2为什么能实现tpc的复用？是因为每个request都加上了一个独立的编号，这样多个request和response就可以对得上。在这里Fastcgi也会需要记录不同请求的id，这是CGI协议中不需要的，因为单进程处理是不用区分的。

FastCGI Progress：同样是执行CGI脚本的进程，与CGI Process的差别体现在fast process这些进程是被管理起来的，可以得到复用，而CGI process是用完就exit

FastCGI模块：相比CGI模块简单的转发、fork进程、回送结果作用，fastcgi模块的作用就要复杂不少，在CGI模块功能的基础上，Fastcgi模块需要管理进程资源池，也就是说每次CGI请求到底要如何合理的分配，这都是Fastcgi模块的功能，而php-fpm就是一个很优秀的fast-cgi模块，同样php-cgi也是这样的一个模块

因为看了好多关于cgi，fastcgi，php-fpm，php-cgi的区别，感觉都不是很清楚，就自己研究了一下，有失误欢迎指出！
