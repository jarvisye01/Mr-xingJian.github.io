探索Git底层工作原理
Posted on 2020-07-04   Updated on 2020-07-04 Git

Git工作原理

本周在git上踩了一个坑，以前对git只是简单地使用，知道几个命令，最多就是暂存区这些概念了，这一次好好地把Git的工作原理梳理一下，加深理解，避免再次出现这样的错误！

Git本质

git的本质是一个内容寻址的文件系统，并在此基础之上提供一个版本控制系统的用户界面，所以我们首先要明白git是一个特殊的filesystem

Git开始的地方

git init是我们用来创建一个git仓库的命令，会在当前目录下新建一个.git目录

  • .git
    • hooks(dir)：包含客户端和服务端的钩子脚本
    • info(dir)：包含一个全局性的排除文件，用以放置.gitignore的忽略模式
    • objects(dir)：存储所有的数据内容
    • refs(dir)：存储指向数据（分支、远程仓库和标签等）的提交对象的指针
    • config：包含项目特有的配置项
    • description：进供GitWeb程序使用，暂时不用关心
    • HEAD：指向当前被检出的分支

还有目前尚未创建的index文件，它将会保存暂存区的信息

内容寻址

内容寻址文件系统说明了git的本质，git的核心是一个键值对的数据库，依托于objects目录，简单点说，你想git仓库插入任何内容value，git会给你返回一个key，通过这个key你可以再次取出value

Git对象

向git仓库中添加的每一个文件都会被转为一个git object对象然后存储

git hash-object -w filename：这个命令会将我们的文件转为一个git object然后存储到objects目录之中，这个命令会返回一个长度为40的十六进制SHA-1哈希值，这个哈希值是通过一个头部信息加上文件内容计算而来的，哈希值的前2位会用作存储这个git object的目录，剩下38位作为文件名

git cat-file -p hash：这个命令会从数据库(/objects目录)中取出对应的数据，并自动判断文件类型显示内容

blob object：上述我们的文件转为的git object成为数据对象，git cat-file -t hash可以告诉我们存储的是什么类型的对象

tree object：有了blob object我们的文件就有了归属，但是我们的目录还没有一个很好的表示，git中用tree object来存储目录的相关信息，tree object中主要存储指向blob object和tree object的指针

  • git update-index –add filename：这个命令会将我们的文件创建一个单独的暂存区
  • git write-tree：这个命令可以将暂存区的内容写入到一棵对象树

commit object：现在你有了这么多的hash值，你需要把他们都记住是不可能的，于是利用commit object来保存这些信息

  • git commit-tree hash：这个命令为一棵tree创建一棵commit tree
  • git log hash：这个命令可以展示我们的提交历史

Git引用

以上我们在使用git log查询提交日志的时候需要知道起点commit tree的hash，这显然不是人脑能够胜任的工作，还记的开始提到的refs文件夹吗？我们可以创建简单好记的引用来帮助我们记忆。

  • refs(dir)
    • heads
    • tags

echo hash > ./git/refs/head/master：我们可以将一棵commit树的hash写入refs/head中的一个文件，这个文件的名字就可以充当这棵树的引用，当然你也可以不叫master

git update-ref refs/head/name hash：我们更多的是使用这个命令来更新引用值，而不是直接修改文件

Tag object

上文提到的git引用是指向commit tree的引用，一般我们会用指向不同commit tree的引用来作为我们的多个分支，但是如果我们想关键指向普通对象的引用该怎么办呢？

这个时候就需要用到tag object，它会指明一个标签的创建者信息、一个日期、一段注释，最重要的是一个指向对象的指针，更多的时候这个指针不会移动，只是给对象取了一个友好的名字

远程引用

如今我们用到git再也不是简单的只是在单机上使用，我们会将代码放在远程的托管平台，例如github、gitlab、gitee等等

这样我们可以添加一个远程版本库，将本地的分支推上去，这时候refs/remotes目录会被创建，然后来记录最近一次推送的分支

这里可以使用git add remote命令添加远程版本库，这里就不演示了，注意远程分支都是只读的，不能通过commit来管理远程分支

引用规范

我们可以通过git add remote origin …命令来添加一个一个远程版本库，远程分支到本地分支的映射也是需要研究一下的

一般git add remote会自动生成配置规则，放在.git/config中

url代表了远程版本库的地址，fetch代表的是映射规则，这代表了远程库refs/heads中的分支都会映射到本地的refs/remotes/origin中，也可以任意配置规则达到你的需求

除了拉取分支你还可以配置相关的推送规则

例如：push = refs/heads/master:refs/heads/test/master，这样你每次都能把master分支推到远程的test/master分支

也可以使用git push origin master:refs/heads/test/master完成

删除分支，就是将远程分支置空：git push origin :[branchname]

总结一下

来总结一下我们的日常概念是如何对应git底层的一些东西的

repository git
file blob object
directory tree object
commit commit tree
refs/heads/ 每一次提交的commit tree
refs/tags tag object
HEAD current commit tree

下面用一个大的example来看看git的工作方式，主要步骤如下：

  1. 创建一个新的repository
  2. 添加一个test1.txt文件，内容为test1，提交，
  3. 添加一个test2.txt文件，内容为test2，提交
  4. 修改test1.txt为test1 modify，提交
  5. 添加文件夹dir，里面添加两个文件，分别是test3.txt => test3，test4.txt => test4
  6. 最后看一看整个提交过程，以及这4次提交之间的关系
  7. 新建一个branchtest分支
  8. 添加test5.txt => test5并提交，看看会改变什么
