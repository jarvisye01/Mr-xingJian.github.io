与Docker的第一次亲密接触
Posted on 2020-07-18   Updated on 2020-07-26 Docker

初探Docker

Background

上一次写了Docker跟虚拟机的一些差别，这段时间也是一直在看Docker相关的一些知识，但是这玩意儿确实还是需要很强的操作系统的知识，Docker本质还是利用了linux操作系统本身的一些优秀特性来运行的。所以很多东西看着看着就跑到操作系统里面去了，还是比较深，这部分自己还是不够扎实，是未来的一个需要努力的方向。

这一次主要是动手试了一试Docker命令，算是一个初体验吧。主要是做了两个小demo，使用基础的centos镜像构建一个包含node环境大的镜像和使用以alpine为基础的golang环境的镜像运行一个简单的rpc服务。

这两周手上的事情比较多，一直在搞业务相关的东西，这些知识的学习没有进一步的推进，人也有点懒散。秋招来了，自己还有很多不够的地方，希望能给自己一点压力，继续前进！

使用Docker commit构建一个node镜像

node是一个Javascript的运行平台，虽然好久没做前端了，但是这一次还是选择了安装node环境来练手，就当是怀恋一下吧。

相关.tar.xz文件的解压，.tar.xz是一个常见的压缩文件，是先用tar命令进行打包，然后使用xz命令对单个文件进行压缩。所以在解压的时候顺序是反过来的，注意的一点事xz是只能压缩文件的，对于目录不起作用，所以需要使用tar命令先进行一个打包的操作。因为之前见的比较多的是.tar.gz文件，这里记录一下.tar.xz文件的一个处理方式。

使用docker commit创建一个镜像，docker commit命令会将我们目前容器中的一切修改记录下来，然后生成一个新的镜像。

本来就是从一个很大的基础镜像上构建的，会导致镜像迅速膨胀

使用Dockerfile构建镜像

这一次不再使用大的基础系统镜像构建，只选择与我们相关的基础镜像构建，alpine是一个很基础的linux镜像，其大小只有10M左右，非常精简。去dockerhub选择一个以alpine为基础的带有golang环境的镜像。

这里我们希望构建一个包含我们所写的rpc服务的一个镜像，以这个镜像为基础运行一个容器，然后提供rpc调用服务，也是一个很简单的流程，主要是为了熟悉相关的知识。

实现一个GO的rpc服务，之前博主有讲过go的net/rpc包中的主要内容，可以回去看一下。

go build构建成相关的二进制文件，现在有了二进制文件，如果我们想要使用容器运行这个rpc服务，那么我们只需要构建一个包含GO环境的镜像就可以了（这其实体现了go的简洁，使用GO似乎我们就真的“省去”了web server这个流程，这是因为go本身帮我们做了很多的工作，web server还是很值得了解一下的），现在有两种方式：

  • 使用一个基础镜像例如centos，运行一个容器，去容器中手动安装GO的环境，然后docker commit构建包含GO环境的镜像，就像上面构建node的环境一样
  • 使用官方包含GO环境的基础镜像，这类镜像更加精简，很适合我们的微服务，选择基于alpine的集成了GO的镜像，这个镜像会小一点（原生的alpine非常小，但是这里要集成GO的环境，也会膨胀）

来分析一下，为什么说Docker的镜像是分层构建的：

  • 首先是上下文：构建的时候会将你指定的目录上传到Docker的daemon进程也就是server端，由server完成构建，所以一旦上下文中的文件过多会需要上传很长时间

  • 运行容器：docker run -d -p 8000:8000 –name rpc_server my_rpc:v1.0

    这里遇到一个问题，容器启动的时候出现错误：standard_init_linux.go:211: exec user process caused “no such file or directory”，很多文章说是什么脚本字符格式问题，但是显然不是，因为我就是在centos上用vim编写的Dockerfile，这里其实是alpine镜像的问题，alpine要求GO的build出的文件必须是静态可执行文件，这里我们使用go build -tags netgo来构建镜像

  • 使用docker ps查看运行的容器

镜像并不是一个文件，更多的时候镜像指的是一组文件，例如删除一个文件并不是真正的删除了而是新建一层镜像标记该文件已经删除，最后容器跑在最上面的一层读写层，寻找文件的时候从上至下寻找

小小的Docker实践

background

我平时比较喜欢使用markdown工具，有时候免不了需要一些图片内容。大家知道markdown插入图片主要是有本机文件链接或者是网络图片地址链接，一种比较主流的形式就是在md文件的目录下创建一个目录存储图片，但是这样的方式总是会带着这个文件夹，一旦你把markdown文件传给别人之后就没用了。

所以我平时都是选择的第二种形式，那就是使用网络文件链接。主要的思路就是使用nginx充当静态内容分发服务器，这种方式的缺点就是如果没有网络就玩不了了，但是估计网络没了你就什么也不想干了，所以这种方式还是可以的。是你的工作环境更加干净，同时实现markdown文件的共享也更加方便了。

初步实现

初步的时候我是直接在服务器上安装nginx，在本机与服务器实现图片传输我是用的是git，我先用git将内容上传到gitee，然后在服务器上使用git拉取。利用nginx对外开放这个目录的内容，这样我就能用网络链接获取我的图片，这样的话其实图片不仅可以使用在markdown文件中，放在网页中也是很方便的。

技术升级

现在有了Docker，我想使用一个容器去帮我实现分发的内容，其实我不仅仅有图片需要分发，之前的一些思维导图也是通过这种形式分发的。所以我想构建一个镜像，这个镜像包括基础的nginx服务，并且我不需要将数据放到容器里面，而是利用数据卷挂载的形式将服务器本机的目录挂载到容器中去。这样我就不必每次都进入容器，而是只需要在服务器中git一下就可以获取最新的资源。

于是我的一次小小升级就决是定这样了，构建一个镜像，这个镜像主要包括nginx，帮助我进行内容分发的工作。当然这已经是一种比较简单的实现了，我们将静态内容分发的任务隔离出来了，将就用一下其实也可以。但是考虑到我每次从本机上传图片到gitee之后还要去服务器更新，这就有点难受了，如果可以搞一个CI工具来帮我们完成持续集成，那么我就可以不用手工去服务器上git pull了，这样就方便多了。

.dockerignore中添加了/source，这是为了我们不让数据直接进入容器，而是后期通过数据卷挂载的形式挂到容器上

构建镜像

执行构建命令：docker build -t sweetcoder/centos_nginx_server:v1.0 .

构建之后顺便推到dockerhub：docker push sweetcoder/centos_nginx_server:v1.0

运行容器

运行容器命令：docker run -d -p 8080:8080 -p 8081:8081 –name nginx_server -v $PWD/source:/var/www/source sweetcoder/centos_nginx_server:v1.0 nginx

By the way

虽然只是几个很简单的步骤，但是实际没有这么顺利，中间经历了几次Dockerfile的内容写错，另外最后还出现了一个乌龙，我在请求图片的的时候又加了一次文件路径，导致出现了404错误。刚开始百思不得其解，以为是构建的过程中出现了什么问题，但是nginx也是正常启动的，最后想到了去容器中看一下nginx的日志，结果恍然大悟，请求报错是no such file，我一看路径重复了两次就顿时明白了。

所以有时候还是需要一些耐心去调试，特别是刚开始学习一门新的内容的时候！
