redis二三事（二）
Posted on 2020-12-04   Updated on 2021-05-24 redis c

字符串和数字

面试的时候常常会被问到redis的数据类型，首当其冲的自然就是string和数字，那么你知道string和数字在redis的底层是怎样实现的吗？

string

要想回答string相关的内容你首先要知道C语言中的一个字符串是怎么表示的，C语言的字符串由一组字符加上’\0’结尾，一般我们可以用一个char类型的指针指向或者是数组存储我们的字符串。对于字符串的存储我们一定要有一个边界意识，就是我们要能够判断一个字符串到什么地方截止，为了体现这种边界，其实有两种方法：

  • 用一个边界字符结尾：例如C语言中用’\0’表示
  • 存储字符串的长度：例如GO语言

二者都有自己的优缺点，比如方法一可以很好地节省内存，但在取字符串长度时的效率并不如方法二，另外方法一表示的字符串并不是二进制安全的，因为一串二进制数据中可能包含’\0’，这时候就没办法很好地判断整个string的边界。方法二的取长度效率非常高，但是需要占用额外的内存来存储长度，对于长度很短的字符串这种空间换时间可能并不是很划算，因为有专门的长度来表示边界，所以方法二中的字符串可以存储任何字符。

在这里redis采用了第二种方法表示字符串，但是却仍然保持了以’\0’为一个string的结尾，你可能会觉得这是一种多余的操作，这样做其实是为了可以使用丰富的C字符串处理函数，#include <string.h>你一定不会陌生吧？

number

整数的实现实际上比较直观，因为整数的长度是跟类型相关的，就是那么几个长度而非常规整。我们可能常常会看到int8_t int16_t int32_t int64_t这些整型类型，存储不同类型的整型数据需要的字节空间是不同的。这样就会出现一个场景，我们可以申请一个int64_t数组来存储所有的数字，可是如果我们的数字都很小，int8_t足以存储那不是浪费了很多的空间？确实是这样，所以redis并没说用空间最大的int64_t来存储所有的数字，而是才用了跟一种非常灵活的数据加编码的形式确定整型的类型。

sds[sds.h sds.c]

sds模块实现了redis中的动态字符串类型，这种字符串可以灵活的伸缩长度，提供空间预分配功能，在我们表示的字符为C风格字符串时可以使用C中的标准字符串处理函数来处理。增强了C字符串的灵活性和安全性，sds在redis内部被广泛的使用。

这里我们要搞清楚的是在redis中虽然定义了sdshdr这个结构，但是最常用的还是typedef char *sds这里定义的sds结构，一般会将sdshdr中的buf赋值给sds，这样的话我们其实就能直接通过sds变量来操作字符串，而不是中间还隔了一层sdshdr，如果我们真的需要相应的sdshdr中的内容，通过buf(sds)指针的位置也是可以直接获取sdshdr指针，你可以看看上面的第二个macro。

sdsnewlen

我来梳理一下具体的步骤：

  • 根据你需要初始化的string的长度initlen来确定需要什么sdshdr来容纳，例如sdshdr8管理的string的最大长度是受到uint8_t的最大值的限制的
  • 根据确定的sdshdr type来获取该header的长度，显然sdshdr8和sdshdr16的结构本身长度是不一样的
  • 根据header长度和string长度来申请内存，注意这里的内存是一起申请的，也就是存储string的内存是仅仅跟在header之后的
  • 根据相应的sdshdr type设置flags等等字段
  • 最后如果我们提供了初始化的内存，复制到我们为字符串申请的内存

这里我们注意一下s = (char*)sh+hdrlen这一段的意思，这样的话我们就是直接在使用sdshdr中的变长数组，也就是所存储的string的内存，最后我们还给字符串的最后加上了’\0’，让它成为一个C风格的字符串，在这里你应该能从代码的层面直观地感受到string的内存是紧跟着sdshdr的。

你也可以想一想为什么需要这几种sdshdr，因为在某一个特别的场景中如果我们需要处理的字符串非常短，例如配置文件的各项配置，那么sdshdr64就会比sdshdr8要浪费很多空间！

sdsfree

释放sds的内存，也很简单，但是需要注意的是sds(char *)并不是我们申请的这一段内存的开始的位置，上面我们已经看到过了，sds实际上是buf的位置，内存开始的位置还个了一个header的大小，这段代码也不是很难理解。

sdsMakeRoomFor

为一个sds增加更多的空间，这个函数常常用在需要将string变长的操作之上，例如常见的concate操作等等。

这里一定要注意到一个问题，也就是在一个sds增加长度之后，原始的sdshdr是否还能满足string的长度要求，如果增长之后旧的type装不下就要更新sdshdr。

这里还有一个重要的问题就是空间的预分配，虽然我们给出的增加的长度是addlen，这是redis并没有老老实实只是将长度增加这么多，在空间足够的情况下：

这种预分配是为了之后每次增长都需要进行realloc操作，我们知道realloc操作是有可能需要整体原来的内存块中的内容的，预分配可以防止连续增加长度时每次都要调用realloc的情况。

sdsRemoveFreeSpace

前面的空间预分配实际上是在内存充足的时候用空间来换取时间的典型做法，但是一旦出现内存不足的时候我们自然就需要对策略进行调整，redis依然拥有收回这部分预分配出去的内存。

空间紧缩的时候也是要注意，长度减小后之前的sdshdr type还能不能适用，这决定我们到底是利用realloc紧缩还是需要重新构建sdshdr。

说完了空间预分配和紧缩之后我们还是再来说一说sdshdr中是怎么管理已经使用的空间和剩余的空间的，首先我们来对sds的内存分成三个段：

  • header
  • string
  • null terminator
  • rest/available(maybe)

sdshdr中有两个跟长度有关的字段，len和alloc，那么len实际上就是指string的长度，alloc实际上是整个malloc申请的内存减去header和null terminator，那么rest/available = alloc - len自然就是表示我们预分配的空间，这样应该很好理解了。

这里关于sds只是说到这里，对于redis其实还是提供了很多的API，但是我觉得主要的核心还是你需要了解sds的内存的布局，不同sdshdr type之间的切换，其他的其实就跟我们的普通string差不太多，大家可以自己去研究剩余的sds代码。剩下的API主要包括：

  • concate：讲一个sds风格或者是C风格的字符串接到一个sds的后面
  • sdscpy：参考strcpy
  • sdsll2str：long long int => sds
  • sdsstr2ll：sds => long long int
  • sdscatvprintf：参考format function in C
  • sdstrim：参考strtrim
  • sdscmp：参考strcmp
  • ……

这里就不一一列举，很多都是string的标准API的sds风格的实现，你能够用到的字符串操作sds基本上都支持！

intset[intset.h intset.c]

有了string的基础，intset其实还是很好理解的，intset顾名思义就是一个整数集合，一般来说我们可能就是用一个array来表示，但是遇到的问题就是这个array的类型是不好确定的，int8_t节省内存，但是表示的范围非常有限，int64_t表示的范围很大，但是浪费内存。也就是说一个固定类型的array是很难满足我们的需求的，接下来就让我们看一看redis中是如何构造intset来解决这个问题的，怎么让intset集合的类型变的可变。

非常简洁，encoding自然就是类型，也是我们的intset中的整数集合类型，这里要额外提一下的是你一定要习惯这种encoding + data（编码 + 数据）的表示方式，这是redis中管用的一种方式。length就是intset的长度了，所以对于redis中的intset获取长度的时间是O(1)。contents是存储具体数据的地方，虽然类型是int8_t，但是并不代表里面的我们的intset是int8_t类型。

_intsetValueEncoding

这个内部模块的函数，主要作用是获取一个数字的最合适的编码，尽量让存储数字的内存空间最小，这以下三个范围：

  • [INT16_MIN, INT16_MAX]
  • [INT32_MIN, INT32_MAX]
  • [INT64_MIN, INT64_MAX]

intsetSearch

从函数名字并不难理解，这个函数就是为了从intset中搜索一个数字，这仅仅是一个非常平常的操作，但是intset中有两个特殊的点：

  • intset中的元素是按照顺序存储的，所以搜索用的是binary search
  • 如果并没有搜索到，会根据二叉搜索的最后的位置，告知调用者被搜索的元素可以插入的位置（插入之后是有序的），这个信息被填入*pos中

intsetUpgradeAndAdd

如果你的intset中本来都是int16_t类型的整数，但是现在出现了一个数字需要插入，并且这个数字的范围在[INT16_MIN, INT16_MAX]之外，这个时候整个intset的编码类型就需要提升，之前的元素的类型都需要完成提升到int32_t，这样的话需要扩充内存，然后执行数据插入。

intsetRemove

从intset中删除一个元素，远远比插入要简单，因为remove方法并没有考虑移除一个元素导致整体编码可以减小的情况，例如一个编码为int32_t的intset中移除最后一个int32_t数字之后并不会考虑整体缩减内存。

redis做的仅仅是将这个元素剔除出去，然后进行一次整体移动即可。
