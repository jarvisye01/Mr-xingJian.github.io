漫谈http
Posted on 2020-08-28   Updated on 2021-05-24 http

与http有关的知识点

http status code

http的状态码应该是最基础，但也是非常重要的知识点，平时我们可能不会很重视，通常200，3XX，4XX，5XX我们可能可以说出一个大概，但是真要深入研究一下的话我们可能又会说不出来，所以这里整理了一下详细的http状态码的相关知识。

1xx

100：很少出现，表示目前一切正常，client继续请求即可。客户端需要在首部的实体之前添加Expect：100-continue

101：表示server收到client升级协议的请求，正在切换协议。例如你要使用WebSocket协议，就需要将http升级，升级的的协议会在response的Upgrade中展示

103：和Link这个header首部一起使用，允许client在server还在准备相应数据的时候预加载一些资源

2xx

200：这个最常见了，就是代表我们的http请求成功了

201：表示请求成功，并且创建了一个新的资源，会返回新建资源的路径，一般用于POST方法

206：表示服务器处理了部分GET请求，常用于与Range配合使用进行断点传输

3xx

300：被请求的资源有一系列可以选择的对象，用后或者浏览器自行选择

301：被请求的资源已经被永久重定向到新位置，搜索引擎可以根据这个更新搜索信息

302：被请求的资源被临时重定向到新位置

304：被请求的资源没有改变，常用于缓存验证的响应状态码

307：被请求的资源被临时重定向

308：被请求的资源被永久重定向，但是接下来的请求方法必须与上一次相同

4xx

400：用户的请求有误，例如参数错误

401： 请求未授权

403：服务器已经解析请求，但是拒绝执行，例如用户想访问一个静态文件夹，但是我们没有开放这个权限就会出现403

404：请求失败，找不到相应的资源

405：请求的方法不允许，例如你写的POST接口被GET来请求

406：请求的资源特性无法满足，例如你的Range范围超出了资源的范围

5xx

500：服务器内部错误，例如你的接口逻辑写错了

501：服务器不支持这个类型的请求

502：网关错误，表明这个服务器作为网关得到了错误的响应

503：服务器没有准备好，例如服务器因为维护停机

504：网关超时，这个服务器作为网关但是不能及时得到上游的响应

Http Cache

缓存是web技术中的关键一环，缓存可以帮助我们在资源请求的某个过程中将资源暂存起来，之后获取资源不用每一次都去到server。理论上来说缓存可以存在于http请求经过的任意地方，从你的browser到中间的代理，再到最终的server，缓存几乎无处不在，另外一些常见的缓存位置还包括网关、CDN等等。

http的缓存通常分为两种：

  • 私有缓存：这种缓存仅仅用于单个用户，例如你的browser中的缓存
  • 共享缓存：这种缓存对所有用户均有效，例如代理服务器上的缓存

在http这个范围之内，通常是对GET请求的response进行缓存，这个缓存的key是与请求的方法和请求的路径有关系。

Cache-Control（如何控制缓存）

http支持用于缓存控制的header，就是Cache-Control字段，这个字段可以有以下取值（策略）：

  • no-store：不执行缓存，每次的请求均去到server
  • no-cache：使用缓存，但是每一次都要验证缓存的有效性，如果有效server返回304（上图就是此种策略）
  • public：中间人可以缓存允许的内容，例如代理和CDN可以缓存部分内容
  • private：中间人不能缓存，只有browser可以缓存
  • max-age：缓存的有效时间
  • must-revalidate：必须验证缓存的有效性

这些字段可以用于client与server之间的缓存协商，达成缓存策略之后可以大大提高请求速度！

Cache Validate(如何验证缓存)

既然是缓存，必然会有过期的时间。一旦缓存过期我们就要重新获取资源，用户刷新之后就会开始验证缓存，通过上述的must-revalidate可以触发缓存验证，另外浏览器中设置Advanced->Cache也能强制验证缓存。

缓存的验证主要有以下几种形式：

  • ETags：server响应会携带一个ETags标签，client下一次请求的时候利用If-None-Match带上Etags到server，server进行验证
  • Last-Modified：server响应会返回一个资源上一次的修改时间，client下一次请求时利用If-Modified-Since带上这个时间，server进行验证

Vary 响应

用户请求可以加一个Vary的header字段，只有当本次请求的Vary字段与缓存的请求和响应的Vary字段时候才会使用缓存。

这个对使用缓存做了更精细的划分，你的移动端和电脑端缓存的资源不同，那么可以使用一个Vary: User-Agent来精细区分！

Http Header

http header主要是用于传输过程中表示某些附加信息，有些请求头是request和response通用的，另外一些则是仅仅在request或者是response中使用，header通常被分成一下几个类别：

  • General header：可用于request和response
  • Request header：包含更多的是client和request的信息
  • Response header：包含更多的是server和response的信息
  • Entity head：包含与传输的实体有关的信息

另外还需要注意的是有些header是端到端的，也就是说中间的代理必须原样转发，不可修改，还有一种叫做逐跳消息头，仅仅对于单个连接有效，不会通过代理缓存或者是转发！

Http 跨域

首先我们先要搞清楚跨域到底是什么？

跨域全称跨域资源共享，使用额外的http header来说明运行在一个domain上的web app可以访问不同domain上的资源。这种不同domain之间的资源请求就是一个跨域的http请求。

怎么确定一个域？

总的说来主要是三个部分，协议、域名和端口，只有这三者完全相同时才是同一个域！例如http://www.test.com/test1.txt 、http://www.test.com/test2.txt 就位于同一个域，http://www.test.com:8080/test1.txt 就处于不同的域，因为后者与他们的端口不同。

常见的跨域请求：

  • 例如html中img标签利用src引用其他域上的资源
  • 脚本发球跨域资源请求，例如你在前端开发的时候fetch后端的接口

这种限制决定了我们在fetch跨域资源的时候是不能正确获取的，我之前在开发的时候通常前端跑在localhost:3000之下，后端跑在localhost:8000之下，这就是典型的跨域行为。

针对跨域，http中也包含了一系列的http header用于处理跨域问题：

response headers：

  • Access-Control-Allow-Origin：表示允许来自哪些domain的跨域请求访问资源
  • Access-Control-Expose-Headers：跨域的响应浏览器只能获得部分的headers，这个header与用于存放允许访问的白名单
  • Access-Control-Max-Age：请求的结果能够缓存多久
  • Access-Control-Allow-Credentials：当浏览器的credentials设置为true时是否允许浏览器读取response的内容
  • Access-Control-Allow-Methods：允许跨域的http methods，用于响应预检请求
  • Access-Control-Allow-Headers：允许使用的http header，用于响应预检请求

request headers：

  • Origin：代表预检请求或者是实际请求的源站
  • Access-Control-Request-Method：在预检请求中说明会用到的跨域方法
  • Access-Control-Request-Headers：在预检请求中说明会用到的http headers

坚决不能动口不动手

  • 后端返回数据的时候添加相关的header
  • 直接请求本域，然后用Nginx做代理访问其他的域

今天终于有时间把这个demo完成了，其实demo很简单。一个简单的前端，使用nginx服务器，后端用golang编写。

相关的项目信息如下：

  • 前端：nginx充当服务器，项目所在的域为http://1.15.14.201:3000
  • 后端：使用golang原生http服务，项目所在的域为http://1.15.14.201:9000

这里后端中有3个接口，都是返回一个list供前端显示：

  • /todolist：前端直接fetch（跨域），后端设置res.Header().Add(“Access-Control-Allow-Origin”, “http://1.15.14.201:3000")允许跨域
  • /somedata：前端直接fetch（跨域），后端不做任何处理
  • /nginxdata：前端fetch本域，通过nginx代理到后端

显而易见，1和3方法都可以很好的解决跨域问题，可以看一下展示效果：
