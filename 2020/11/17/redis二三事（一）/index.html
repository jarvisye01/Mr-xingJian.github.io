redis二三事（一）
Posted on 2020-11-17   Updated on 2020-12-06 redis c

Redis

redis是一款非常优秀的开源软件，目前被广泛的应用于缓存、消息中间件等多个领域。Redis完全基于内运行，使用多路复用IO模型，用C语言实现了丰富的数据结构，这些优点使得Redis越来越流行，几乎所有公司的后端都会用到Redis。

这样的市场情况也使得Redis成为面试的高频问点，博主3月份在找实习的时候吃了很大的亏就是我的项目中没有用到Redis，对Redis本身也并不是很了解，被问到Redis相关的就很尴尬。关于Redis其实有很多学习的路线，如果我们做项目业务多一点的话就要更加注重怎么去构建大型的Redis集群，根据业务特点来配置Redis，另外一条路线就是像我这样就是想单纯的学习，在学习学习了OS和各种系统调用之后借助Redis源码来帮助自己进一步的提升。

从这篇Blog开始我会陆续地写一写和Redis源码分析相关的内容，看这种比较大的开源软件的源码我也是第一次尝试，如果有什么写的不对的地方欢迎指出！

这里我主要是从一些Redis源码中的文件开始分析，因为一般一个文件就是Redis中的一个模块。

zmalloc[zmalloc.h, zmalloc.c]

zmalloc模块是Redis对于标准库内存管理函数的一次封装，这个模块还是比较简单的。主要是对malloc、calloc、realloc的封装，并且该模块中设置了一个used_memory的atomic var，这个变量记录了Redis使用内存的数量。这里仅仅用对malloc举例子，剩下来两个函数都是差不多的，先看代码：

这个模块就是对stdlib标准库中内存函数的封装，每个内存函数都有一个附加的usable函数报告使用的内存，关于used_memory原子变量的更新是利用 “atomicvar.h”模块中导出的atomic var操作接口实现的。刚看源码时不要想着每一行代码都搞明白，知道是原子更新used_memory就行了！

ae[ae.h, ae.c]

ae模块主要是Redis事件循环模块，Redis通过单线程不断地轮询各种出现的事件，这其中又涉及了多路复用IO，特地要说明的是Redis会选择你的OS平台之下最优的多路复用IO模型，依次是epoll，poll和select。

既然是时间，那么就要谈一谈Redis中的事件，在Redis中事件分为两种：

  • file-event
  • time-event

通过typedef定义了一系列的函数原型，使得我们之后可以灵活的绑定处理事件的函数，这是函数指针灵活运用的例子，完成了事件与具体处理方法之间的解耦，非常值得我们去学习！

那么Redis是怎么统一管理相关的file-event和time-event的呢？其中还定义了eventLoop这个顶级实例，保存所有的事件。

a small summary

到了这里我们对aeEventLoop就会有一个比较直观的理解，这里结构体用来管理我们的Redis中的两种重要的事件，其中文件事件是用数组进行管理的，时间事件是用链表管理的，每一个aeFileEvent和aeTimeEvent结构里面都保存了相应的函数指针（hook）。

但是文件事件更多的是Redis处理连接的读写请求，这里实际上是用到了多路复用的IO模型，apistate保存了epoll(select, poll)实例，之后文件事件的处理会需要epoll的帮助。另外触发的事件会被放进fired数组，所以之后只需要对fired数组中的文件事件进行相关的处理就行了。

好了，当你看到这里之后就会明白整个事件的管理逻辑，那么后续肯定会提供一系列的管理API，就是我们常说的增删改查，这里我就只是把这些API列出来，不做进一步的解释了，大家可以自己看一下，有些里面还是有一些很细小的逻辑需要理解，但是并不影响整个源码的阅读！

Manage API：

  • int aeCreateFileEvent(aeEventLoop *eventLoop, int fd, int mask, aeFileProc *proc, void *clientData) => 创建一个文件事件放入aeEventLoop
  • void aeDeleteFileEvent(aeEventLoop *eventLoop, int fd, int mask) => 删除一个文件事件中的mask掩码包含的事件
  • int aeGetFileEvents(aeEventLoop *eventLoop, int fd) => 根据fd获取一个文件事件
  • long long aeCreateTimeEvent(aeEventLoop *eventLoop, long long milliseconds, aeTimeProc *proc, void *clientData, aeEventFinalizerProc *finalizerProc) => 创建一个时间事件
  • int aeDeleteTimeEvent(aeEventLoop *eventLoop, long long id) => 删除一个时间事件
  • static long msUntilEarliestTimer(aeEventLoop *eventLoop) => 获取触发时间离现在最近的时间事件

需要说一下的就是这里对文件事件的区分是利用fd(file descriptor)完成的，而对时间事件的区分是利用一个long long类型的id完成的。以上的API的操作无非就是在数组和链表中删除或者是增加一个元素，元素的对比分别利用fd和id完成。（当然，细节大家还是自己看！）

process event

说了整个aeEventLoop是怎么创建，怎么管理各种事件之后自然就是要理解Redis是怎么处理这里文件事件和时间事件的。之前的内容我简单的说过对文件事件的监听是利用相关的多路复用IO完成，触发的事件会放进fired数组，自然就是遍历fired数组完成。对于时间事件的处理我还没有进一步说明，其实也很简单，就是暴力的遍历那一条链表，如果时间到了就触发。

因为时间事件的处理十分简单，对于时间事件的处理被单独抽离成了一个processTimeEvents数组，后面的全部事件处理函数processEvents会调用时间事件处理函数，下面依次来看一下这两个函数。

1.processTimeEvents

所以对于事件时间的处理是很简单的，无非就是将之前标记为AE_DELETED_EVENT_ID的事件从链表删除，对已经达到执行时间的事件调用callback，在调用完成之后有两个选择：

  • 将触发时间推迟（会一次又一次的触发 => 循环事件）
  • 标记为AE_DELETED_EVENT_ID，下一次再删除

2.aeProcessEvents

这里唯一复杂的地方就是 AE_BARRIER会影响事件触发的顺序，我们依次讨论：

  • 设置AE_BARRIER：先处理写事件，然后处理读事件
  • 不设置AE_BARRIER：先处理读事件，然后处理写事件

这里要理清epoll API调用的时间参数的作用，这里的tvp时间参数会造成epoll等待的时间不同，可能等待一定的时间返回，或者是立即返回，又或者是等待直到有事件发生时就会返回。主体逻辑就是先处理文件事件，然后处理时间事件。

eventLoop start

上面我们说了，整个redis事件处理流程就是先处理文件事件，再处理时间事件的一个顺序，我们肯定需要一个让这个事件循环的对象运行起来的入口，这就是aeMain函数的作用了。

其实很简单，就是不断地循环，调用aeProcessEvents去处理eventLoop这个对象中的事件，很好理解。

到这里，其实整个事件循环模块就已经介绍完了，当然我们只是弄懂了大致的一个运行逻辑，更深的一些细节（比如那些处理事件的标志有什么作用，增删事件的一些细节，以及每次处理事件的时候如果又有新的事件加入该怎么处理）没有介绍，这个还是要大家一行一行的去理解相关的代码，这里限于博主个人的精力就不在继续展示，所有的解答都在代码中！

epoll[ae_epoll.h, ae_epoll.c]

上面在讲述事件循环模块的时候我提到了对于file-event的处理跟网络有关，再处理很多连接的时候用到了多路IO复用这个手段来提升redis的性能。redis事件循环是一种经典的单线程多路IO复用模型，这里我只介绍epoll这种模型的使用方法，然后分析redis是如何使用的。

首先我们要了解epoll的工作原理，在常见的web服务器模型中我们一般是给每一个客户端分配一个连接，但是在客户端与服务器通信的过程中的IO是会阻塞的（例如服务端可能要花很长的时间来等待客户端传来的数据），这样就会导致每个连接的性能全部消耗在了IO上面，并没有很好地发挥CPU的计算能力。于是人们就想到了一种方法，能不能让一个进程（线程）来监听我们感兴趣的套接字描述符，等到其中出现了可以进行读写的时候就返回，我们的工作进程（线程）就可以直接处理，而不用一直等待。这样的结果就是在连接很多的时候（套接字描述符），只需要一个进程去监听这一组描述符，工作进程暂时进入休眠状态，等到有相关的连接（套接字描述符）可以读写的时候再唤醒相应的工作进程。

这种监听一组套接字描述符的任务实际上是操作系统的内核帮助我们完成，我们需要调用OS提供的API。另外我们也要意识到，多路IO复用跟单进程多进程没有直接关系，如果我们把调用epoll API的进程成为master进程的话，master完全可以把就绪的套接字描述符分给很多的worker进程处理，或者也可以自己逐个处理，这二者就是多进程和单进程。为了实现的简单和极致的性能，redis采用了单进程模型。（所以我们不能说redis整个就是一个单进程，而是redis在处理事件的时候是一种多路IO复用的单进程模型）。

timeout为0，即使没有事件也会返回0，timeout为-1，一直等待到有事件出现，timeout为整数表示等待一定的时间返回。

关于epoll我们还要知道水平触发和边缘触发：

  • Level-triggered：只要fd中有事件（可读可写）就会返回
  • Edge-trigered：只有fd出现了新事件才会返回

例如我们用epoll监听一个fd，这个fd中被写入了1K数据，这二者都会返回，应为符合他们的触发条件。但是如果只读取0.5K的数据，再次用epoll监听，那么边缘触发会立即返回，而水平触发并不会。

明白了吗？我们将发生了事件的fd放到eventLoop中的fired数组中去，还记得上面的aeProcessEvent是怎么处理的吗？可不就是用我们注册的事件处理函数去逐个处理fired中的fd。

所以aeEventLoop->apistate只是对epoll实例的一个包装而已，利用它可以实现多路复用IO，而且这里redis在等待之后并没有把事件分发给另外的进程，而是就地处理，这体现了redis的单进程！
